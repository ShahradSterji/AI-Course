جلسه ۸: API — از مبانی نظری تا معماری، طراحی، امنیت و پیاده‌سازی حرفه‌ای

تحلیل عمیق پروتکل HTTP، اصول REST، استانداردهای صنعتی، امنیت، قراردادهای داده و پیاده‌سازی مدرن با FastAPI

در این جلسه، API را از پایین‌ترین سطح — پروتکل HTTP — تا سطح معماری، طراحی، الگوهای مهندسی، امنیت و پیاده‌سازی عملی بررسی می‌کنیم. هدف، تبدیل دانشجو از «کاربر API» به «طراح API» است؛ یعنی فردی که قادر است APIهای پایدار، توسعه‌پذیر، امن و استاندارد سازمانی طراحی کند.

بخش اول: API چیست و چرا مهم است؟ — تحلیل مفهومی، معماری و نقش آن در سیستم‌های مدرن

API ستون فقرات دنیای نرم‌افزار مدرن است. اپلیکیشن‌ها، سرویس‌ها، سیستم‌های توزیع‌شده، موبایل، وب‌اپلیکیشن‌ها، IoT و حتی سیستم‌های هوش مصنوعی، هیچ‌کدام بدون API قابل اتصال و تعامل نیستند.

۱.۱. تعریف پیشرفته API

API فقط «رابط برنامه‌نویسی» نیست؛ بلکه یک قرارداد ارتباطی (Communication Contract) است که موارد زیر را مشخص می‌کند:

  • چطور درخواست ارسال می‌شود
  • ساختار دقیق پاسخ‌ها چیست
  • چطور خطاها مدیریت می‌شوند
  • چگونه امنیت، نسخه‌بندی و توسعه‌پذیری کنترل می‌شود

۱.۲. APIها در معماری سیستم‌های مدرن

APIها قلب معماری‌های امروز هستند:

  • Microservices — تمام سرویس‌ها با API به‌هم متصل‌اند
  • Frontend/Backend — فرانت‌اند فقط با API با بک‌اند ارتباط دارد
  • Mobile Apps — کاملاً API-Driven هستند
  • Machine-to-Machine Communication
  • Cloud Services و Serverless Functions

در واقع API همان چیزی است که «اکوسیستم نرم‌افزاری» را ممکن کرده است.

بخش دوم: پروتکل HTTP — زیربنای تمام APIهای RESTful

برای طراحی یک API حرفه‌ای، باید HTTP را نه فقط به‌عنوان یک پروتکل، بلکه به‌عنوان یک «زبان ارتباطی استاندارد» بفهمیم.

۲.۱. ساختار یک درخواست HTTP 


METHOD /resource HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer TOKEN

{ "field": "value" }

هر درخواست شامل سه بخش است:

  • Request Line — متد + مسیر
  • Headers — اطلاعات ارتباطی، امنیتی و متادیتا
  • Body — دادهٔ اصلی (برای POST/PUT/PATCH)

۲.۲. پاسخ HTTP 


HTTP/1.1 200 OK
Content-Type: application/json

{ "status": "success", "data": ... }

بخش سوم: طراحی REST — اصول، الگوها و مسیرهای استاندارد

REST یک معماری ساده نیست؛ یک «فلسفهٔ طراحی» است که API را از سطح یک سرویس ساده به سطح یک استاندارد سازمانی ارتقا می‌دهد.

۳.۱. اصول بنیادین REST

  • Stateless — درخواست مستقل است
  • Resource-Oriented — همه‌چیز یک ریسورس است
  • Uniform Interface — ساختار ثابت برای همه
  • Client/Server Separation — استقلال کامل دو طرف
  • Cacheability — پاسخ‌ها قابل کش باشند
  • Layered Architecture — چندلایه بودن سیستم

۳.۲. ساختار مسیرهای صحیح 

// مسیرهای درست (Resource-Oriented)
GET   /users            → دریافت همهٔ کاربران
GET   /users/123        → دریافت یک کاربر
POST  /users            → ایجاد کاربر
PUT   /users/123        → بروزرسانی کامل
PATCH /users/123        → بروزرسانی جزئی
DELETE /users/123       → حذف

// مسیرهای نادرست — استفاده از فعل در URL (Anti-Pattern)
GET /getAllUsers
POST /createUser

بخش چهارم: امنیت API — مدل‌ها، تهدیدها و معماری دفاعی

امنیت API یکی از مهم‌ترین بخش‌های طراحی است. هر API عمومی یا سازمانی باید تحت یک معماری امنیتی چندلایه اجرا شود.

۴.۱. روش‌های احراز هویت

  • API Key — ساده اما محدود در امنیت
  • JWT — توکن امضاشده، مناسب سیستم‌های مدرن
  • OAuth2 — مناسب سرویس‌های بزرگ و Third-Party

۴.۲. تهدیدها و راهکارها

  • Rate Limiting — جلوگیری از حملات درخواست انبوه
  • Input Validation — جلوگیری از XSS/SQL Injection
  • HTTPS — رمزنگاری ارتباط
  • CORS — کنترل دسترسی کلاینت‌ها
  • Logging & Auditing — ثبت رفتارهای مشکوک

بخش پنجم: استانداردسازی پاسخ‌ها — قرارداد داده و مدیریت خطا

یک API حرفه‌ای، قرارداد پاسخ یکپارچه دارد. یعنی همهٔ پاسخ‌ها — موفق یا خطا — یک ساختار مشخص دارند.

۵.۱. پاسخ موفق 

{
  "status": "success",
  "data": { ... },
  "timestamp": 1700000000
}

۵.۲. پاسخ خطا 

{
  "status": "error",
  "message": "User not found",
  "code": 404
}

بخش ششم: پیاده‌سازی API سطح سازمانی با FastAPI

FastAPI یک فریمورک فوق‌العاده سریع، مدرن و استاندارد برای ساخت APIهای مقیاس‌پذیر است.

۶.۱. نصب 

pip install fastapi uvicorn

۶.۲. پیاده‌سازی API پایه 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    username: str
    age: int

db = []

@app.get("/users")
def get_users():
    return {"status": "success", "data": db}

@app.post("/users")
def create_user(user: User):
    db.append(user.dict())
    return {"status": "created", "data": user}

۶.۳. اجرای سرور 

uvicorn main:app --reload

۶.۴. مستندات خودکار

  • /docs — Swagger UI
  • /redoc — Redocly Documentation

بخش هفتم: قابلیت‌های پیشرفته — فیلتر، صفحه‌بندی، Middleware

در APIهای سازمانی، فیلتر، جستجو، اعتبارسنجی، نسخه‌بندی و Middleware اجباری هستند.

۷.۱. فیلتر داده 

@app.get("/search")
def search_users(min_age: int = 0, max_age: int = 120):
    result = [u for u in db if min_age <= u["age"] <= max_age]
    return {"results": result}

۷.۲. صفحه‌بندی استاندارد 

@app.get("/users/page")
def paginated_users(page: int = 1, size: int = 10):
    start = (page - 1) * size
    end = start + size
    return db[start:end]

۷.۳. Middleware 

@app.middleware("http")
async def logger(request: Request, call_next):
    print("New Request:", request.url)
    response = await call_next(request)
    return response

بخش تکمیلی: تفاوت API با وب‌سرویس‌ها و معماری‌های ارتباطی

یکی از سردرگمی‌های رایج، تفاوت میان API و Web Service است. هرچند این دو گاهی به‌جای هم استفاده می‌شوند، اما از نظر معماری و حوزهٔ کاربرد، تفاوت‌های مهمی دارند.

۱. API چیست؟

هر رابطی که امکان تعامل نرم‌افزاری بین دو سیستم را فراهم کند: از توابع داخلی یک زبان تا سرویس‌های تحت وب. API یک مفهوم «عام‌تر» است.

۲. Web Service چیست؟

وب‌سرویس یک نوع «API مبتنی بر شبکه» است که حتماً از پروتکل‌های تحت وب (HTTP / SOAP / XML-RPC) استفاده می‌کند. یعنی وب‌سرویس زیرمجموعه‌ای از API محسوب می‌شود.

تفاوت‌های کلیدی

  • API یک مفهوم کلی است، اما وب‌سرویس فقط در بستر شبکه و معمولاً اینترنت کار می‌کند.
  • وب‌سرویس‌های SOAP معمولاً از XML و قراردادهای سخت‌گیرانه استفاده می‌کنند؛ APIهای مدرن REST از JSON.
  • API ممکن است اصلاً در وب وجود نداشته باشد—مثلاً API داخلی سیستم‌عامل.
  • REST API سبک‌تر، سریع‌تر، و مستقل از قالب داده است.

بخش نهایی: پروژهٔ عملی — ساخت API خلاصه‌ساز متن با FastAPI و یک مدل رایگان هوش مصنوعی

در این پروژه، یک API واقعی می‌سازیم که متن ورودی را گرفته و با استفاده از یک مدل متن‌باز (رایگان) آن را خلاصه می‌کند.

مرحله ۱: نصب پکیج‌ها 

pip install fastapi uvicorn transformers torch

مرحله ۲: انتخاب مدل رایگان

ما از مدل facebook/bart-large-cnn برای خلاصه‌سازی استفاده می‌کنیم. این مدل کاملاً رایگان و متن‌باز است.

مرحله ۳: کدنویسی API 

from fastapi import FastAPI
from pydantic import BaseModel
from transformers import pipeline

# بارگذاری مدل خلاصه‌سازی
summarizer = pipeline("summarization", model="facebook/bart-large-cnn")

app = FastAPI()

class TextInput(BaseModel):
    text: str

@app.post("/summarize")
def summarize_text(data: TextInput):
    result = summarizer(data.text, max_length=150, min_length=30, do_sample=False)
    return {
        "status": "success",
        "summary": result[0]["summary_text"]
    }

مرحله ۴: اجرای سرور 

uvicorn main:app --reload

مرحله ۵: تست API

ارسال یک درخواست POST به مسیر /summarize: 

{
  "text": "اینجا یک متن طولانی قرار می‌گیرد که باید خلاصه شود ..."
}

مرحله ۶: مستندات خودکار

  • /docs → Swagger UI
  • /redoc → مستندات Redoc

توضیح فرایند

  • API یک متن دریافت می‌کند.
  • متن وارد مدل BART می‌شود.
  • مدل نسخهٔ کوتاه، خلاصه‌شده و منسجم متن را تولید می‌کند.
  • API خلاصه را در قالب JSON برمی‌گرداند.

این پروژه یک نمونهٔ کامل از اتصال FastAPI با یک مدل هوش مصنوعی است—بدون نیاز به هیچ سرویس پولی.
